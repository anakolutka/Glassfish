Novo Cluster

Use a página Novo Cluster para criar um cluster.

A página Novo Cluster contém as seguintes opções.

Nome do Cluster

O nome do cluster.

O nome deve atender aos seguintes requisitos:

  • O nome só pode conter caracteres ASCII.

  • O nome deve começar com uma letra, um número ou um sublinhado.

  • O nome pode conter só os seguintes caracteres:

    • Letras minúsculas

    • Letras maiúsculas

    • Números

    • Hífen

    • Ponto

    • Sublinhado

  • O nome deve ser único no domínio e não deve ser o nome de outro cluster, uma configuração nomeada, uma instância do GlassFish Server ou um nó.

  • O nome não deve ser domain, server, nem nenhuma outra palavra-chave que esteja reservada pelo GlassFish Server.

Configuração

Lista drop-down das configurações nomeadas existentes que o cluster pode utilizar. Um cluster precisa de uma configuração nomeada que defina a configuração de todas as instâncias adicionadas ao cluster. É necessário selecionar uma configuração na lista. O cluster utilizará a configuração selecionada quando o cluster for criado.

Criar uma Cópia da Configuração Selecionada

Se essa opção for selecionada, a configuração selecionada será copiada quando o cluster for criado. Foi designado à cópia da configuração o nome cluster-name-config, em que cluster-name é o nome do cluster digitado no campo Nome do Cluster.

Se default-config estiver selecionado na lista drop-down Configuração, é necessário selecionar esta opção. A configuração default-config pode ser apenas copiada, não referenciada.

Referenciar a Configuração Selecionada

Se esta opção for selecionada, o cluster utilizará a configuração nomeada existente especificada.

Se default-config estiver selecionado na lista drop-down Configuração, esta opção não deve ser selecionada. A configuração default-config não pode ser referenciada, somente copiada.

Tipo de Configuração do Cluster do Message Queue

Use esta opção para especificar o tipo de cluster do broker do Message Queue a ser usado para o novo cluster do GlassFish Server. Por default, GlassFish Server usa um cluster do broker do Message Queue convencional com um broker mestre, com um broker incorporado em cada uma das instâncias no cluster do GlassFish Server.

Se esta opção estiver definida como Personalizado, várias opções adicionais são exibidas para configurar o cluster do broker do &cnMessageQueue para usar o cluster do GlassFish Server. Consulte Opções para Clusters do Broker Personalizados do Message Queue.

Instâncias do Servidor a Serem Criadas

Uma lista das instâncias do GlassFish Server que serão criadas quando o cluster for criado. Para cada instância, as informações a seguir são fornecidas:

Nome da Instância

O nome da instância.

Peso

Um inteiro que representa o peso do balanceamento de carga da instância.

O peso do balanceamento de carga determina a proporção de todas as solicitações do cluster que a instância deve processar.

Por exemplo, em um cluster com duas instâncias, é possível processar uma das quatro solicitações em uma instância e processar três das quatro solicitações na outra instância. Nesta situação, defina como 1 o peso da instância que processa uma das quatro solicitações e como 3 o peso da outra instância. Se preferir utilizar porcentagem, defina os pesos das instâncias como 25 e 75, respectivamente.

O valor default é 100.

Nó

Lista drop-down dos nós existentes nos quais a instância pode estar. É necessário selecionar um nó na lista. A instância residirá no nó selecionado quando o cluster for criado.

As Instâncias do Servidor a serem Criadas também contêm as seguintes opções.

Novo

Botão para criar uma instância. Se clicar neste botão, uma linha é adicionada à tabela Instâncias do Servidor a serem Criadas.

Deletar

Botão para deletar uma instância.

Opções para Clusters do Broker Personalizados do Message Queue

Quando o Tipo de Configuração do Cluster do Message Queue for definido como Personalizado, as seguintes opções serão exibidas para configurar o cluster do broker do Message Queue para usar o cluster do GlassFish Server.

Tipo de Serviço JMS

Os tipos de brokers (chamados hosts JMS no GlassFish Server) a serem usados no cluster do broker. O tipo escolhido determina quais tipos de clusters do broker estão disponíveis. Os tipos de broker são os seguintes:

Incorporado

A configuração e o ciclo de vida do broker são gerenciados pelo GlassFish Server. Cada instância clusterizada do GlassFish é atendida por um broker executado na mesma JVM da instância.

Se a opção Tipo de Serviço JMS for definida como Incorporado, apenas clusters do broker convencionais do Message Queue serão suportados. As opções para configuração do cluster convencional são exibidas.

Local

A configuração e o ciclo de vida do broker são gerenciados pelo GlassFish Server. Cada instância clusterizada do GlassFish é atendida por um broker executado em uma JVM separada no mesmo host da instância.

Se a opção Tipo de Serviço JMS estiver definida como Local, os clusters do broker convencional e aprimorado (altamente disponível) do Message Queue serão suportados. As opções para seleção do tipo de cluster e configuração do cluster são exibidas.

Remoto

O configuração e o ciclo de vida do broker e do cluster do broker são gerenciados usando as ferramentas administrativas do Message Queue.

Se a opção Tipo de Serviço JMS estiver definida como Remoto, nenhuma outra opção será exibida.

Tipo de Cluster JMS

O tipo de cluster do broker do Message Queue a ser usado para o cluster do GlassFish Server.

Se a opção Tipo de Cluster JMS estiver definida como Convencional, as seguintes opções serão exibidas.

Tipo de Armazenamento da Configuração JMS

O tipo de armazenamento de dados a ser usado para os dados de configuração do cluster convencional. Se esta opção estiver definida como Broker Mestre, um broker no cluster será designado como broker mestre e os dados de configuração serão armazenados por ele. Se esta opção estiver definida como Banco de Dados Compartilhado, os dados de configuração serão armazenados em um armazenamento de dados JDBC acessível a todos os brokers. Neste caso, as opções relacionadas ao banco de dados serão exibidas.

Tipo de Armazenamento da Mensagem JMS

O tipo de armazenamento de dados que cada broker usará para armazenar seus dados de mensagem. Se esta opção estiver definida como Arquivar, cada broker armazenará seus dados de mensagens em um armazenamento de dados baseado em arquivo. Se esta opção estiver definida como JDBC, cada broker armazenará seus dados de mensagens em um armazenamento de dados JDBC.

Se a opção Tipo de Cluster JMS estiver definida como Aprimorado (Altamente Disponível), serão exibidas as opções relacionadas ao banco de dados.

Fornecedor do Banco de Dados
URL do Banco de Dados
Usuário do Banco de Dados

O fornecedor do banco de dados, URL de acesso e o banco de dados JDBC a ser usado em qualquer uma destas situações:

  • Quando o tipo de armazenamento da configuração de um cluster convencional estiver definido como Banco de Dados Compartilhado

  • Quando o tipo de armazenamento de mensagens de um cluster convencional estiver definido como JDBC

  • Quando o tipo de cluster estiver definido como Aprimorado (Altamente Disponível)

Para obter informações sobre os fornecedores suportados e os formatos de URLs de acesso para cada fornecedor, consulte PJDBC-Based Persistence em Oracle GlassFish Server Message Queue Administration Guide.

Autenticação do Banco de Dados
Senha do Banco de Dados
Alias da Senha

As informações da senha para o usuário do banco de dados JDBC especificado em Usuário do Banco de Dados.

Se Autenticação do Banco de Dados estiver definida como Senha, a opção Senha do Banco de Dados será exibida. Se Autenticação do Banco de Dados estiver definida como Alias da Senha, a opção Alias da Senha será exibida.

Propriedades do Cluster JMS

Uma lista de uma ou mais propriedades do broker do Message Queue para configurar o brokers. A lista é separada por dois-pontos (:) e tem o formato:

prop1Name=prop1Value:prop2Name=prop2Value:...

Se uma propriedade do broker incluir pontos, insira duas barras invertidas antes dos pontos (\\); for exemplo, para incluir a propriedade imq.system.max_count, especifique imq\\.system\\.max_count na lista.
