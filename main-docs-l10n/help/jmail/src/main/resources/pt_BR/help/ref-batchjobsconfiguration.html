Configuração do Tempo de Execução em Batch

O GlassFish Server fornece um tempo de execução em batch para a programação e execução de jobs em batch. Os jobs em batch geralmente são de longa execução, tarefas orientadas a alto volume que contêm uma série de etapas. Aplicações em lote submetem jobs ao tempo de execução em batch e fornecem instruções sobre como e quando executar as etapas. O tempo de execução em batch processa as etapas e armazena informações sobre os jobs em um repositório de jobs. No GlassFish Server, o repositório de jobs é um banco de dados.

Use a página Configuração de Tempo de Execução em Batch para exibir e alterar a configuração do tempo de execução em batch. O tempo de execução em batch usa um serviço executor gerenciado e uma origem de dados para executar jobs em batch. O serviço executor gerenciado fornece threads aos jobs, e a origem de dados armazena informações sobre o job. Os dados de configuração de runtime de batch são armazenados no elemento config no domain.xml.

A página Configuração do Tempo de Execução em Batch contém as seguintes informações:

Nome da Instância
Nome do Cluster

O nome da instância de cluster ou servidor do GlassFish à qual a configuração se aplica. Este é um campo somente para leitura.

Nome de Pesquisa do Serviço Executor

O nome da pesquisa JNDI do serviço executor gerenciado que fornece threads a jobs batch. O serviço executor gerenciado default é concurrent/__defaultManagedExecutorService.

O serviço executor gerenciado pode ser alterado após um job em batch ter sido submetido ao tempo de execução em lote sem causar impacto sobre a execução do job.

Nome de Pesquisa da Origem de Dados

O nome da pesquisa JNDI da origem de dados que armazena informações sobre jobs em batch atuais e anteriores. A origem de dados default é jdbc/__TimerPool.

Não altere a origem de dados após o primeiro job em batch ter sido submetido à execução do tempo de execução em batch. Se a origem de dados tiver de ser alterada, interrompa e reinicie o domínio e, em seguida, faça a alteração antes de qualquer job ser inicializado ou reinicializado. No entanto, depois que a origem de dados tiver sido alterada, as informações armazenadas na origem de dados anterior ficarão inacessíveis.
