Modifier un module d'écoute HTTP

  1. Dans l'arborescence de navigation, développez le nœud Configuration.
  2. Sous le nœud Configuration, développez le nœud Service HTTP.
  3. Sous le nœud Service HTTP, sélectionnez le nœud Modules d'écoute HTTP.
  4. Sur la page Listeners HTTP, cliquez sur le nom du listener que vous modifiez.

    Si vous modifiez le listener HTTP spécial nommé admin-listener, vous devez redémarrer le serveur pour appliquer les modifications. La Console d’administration n'identifie pas qu'un redémarrage est requis dans ce cas.

  5. Cochez la case Listener activé pour activer ce listener à chaque redémarrage du serveur.
  6. Cochez la case Sécurité activée pour sécuriser ce listener.
  7. Dans le champ Adresse réseau, entrez l'adresse IP du serveur.

    Si le module d'écoute doit surveiller toutes les adresses IP du serveur, entrez 0.0.0.0 dans le champ Adresse réseau.

  8. Dans le champ Port du module d'écoute, entrez la valeur du port pour ce module d'écoute.
    • Si vous avez saisi 0.0.0.0 à l'Étape 7, assurez-vous que la valeur de port du listener n'est pas déjà utilisée par une adresse IP pour le serveur.
    • Sinon, assurez-vous que la valeur de port du module d'écoute n'est pas utilisée par l'adresse IP que vous spécifiez.
  9. Dans la liste déroulante Serveur virtuel par défaut, sélectionnez un serveur virtuel.
  10. Dans le champ Nom du serveur, entrez le nom d'hôte à utiliser dans les URL envoyées au client par le serveur.

    Ce nom est l'alias si votre serveur en utilise un. Si votre serveur n'utilise aucun alias, laissez ce champ vide.

  11. Pour réacheminer des requêtes vers un autre port, saisissez une valeur dans le champ Port de redirection.

    Enterprise Server réachemine automatiquement la requête si les deux conditions suivantes sont remplies :

    • Ce listener prend en charge des requêtes non SSL.

    • Une requête pour laquelle une contrainte de sécurité correspondante requiert le transport SSL est reçue.

      Par défaut, Enterprise Server utilise le numéro de port spécifié dans la requête d'origine.

  12. Dans le champ Threads accepteur, entrez le nombre de threads pour ce module d'écoute.
  13. Désélectionnez la case à cocher Powered By activé pour désactiver l'ajout de l'en-tête X-Powered-By: Servlet/2.4 aux en-têtes de réponse HTTP générés par le servlet.

    La spécification Java Servlet 2.4 définit cet en-tête, que les conteneurs peuvent ajouter aux réponses générées par le servlet. De même, la spécification JavaServer Pages (JSP ) 2.0 définit un en-tête X-Powered-By: JSP/2.0 à ajouter (si vous le souhaitez) aux réponses utilisant la technologie JSP. L'ajout de l'en-tête X-Powered-By: JSP/2.0 est activé par défaut pour les applications Web. Ces en-têtes ont pour objectif d'aider les administrateurs à collecter des données statistiques sur l'utilisation des technologies Servlet et JSP.

    Pour plus d'informations sur l'activation et la désactivation de l'en-tête, X-Powered-By des fichiers JSP, reportez-vous à l'Annexe A, Deployment Descriptor Files du Sun GlassFish Enterprise Server v3 Prelude Application Deployment Guide.

    Il est possible que les environnements de production omettent volontairement la génération d'en-têtes X-Powered-By pour masquer leur technologie sous-jacente.

  14. Cochez la case Blocage activé pour activer le blocage de ce module d'écoute et des ports externes.
  15. (Facultatif) Dans la section Autres propriétés, spécifiez d'autres propriétés.
  16. Cliquez sur Enregistrer.
