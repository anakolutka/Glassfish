Créer un serveur virtuel

  1. Dans l'arborescence de navigation, développez le nœud Configuration.
  2. Sous le nœud Configuration, développez le nœud Service HTTP.
  3. Sous le nœud Service HTTP, sélectionnez le nœud Serveurs virtuels.
  4. Dans la page Serveurs virtuels, cliquez sur Nouveau.

    La page Nouveau serveur virtuel s'ouvre.

  5. Dans le champ ID, entrez une valeur identifiant le serveur virtuel en interne.

    Cette valeur n'est pas visible pour les clients HTTP. Les noms d'hôtes visibles des clients HTTP doivent être spécifiés dans le champ Hôtes.

  6. Dans le champ Hôtes, entrez les noms d'hôte ou les adresses IP de la machine sur laquelle le serveur s'exécute.

    Utilisez des noms d'hôte réels ou virtuels enregistrés dans le serveur DNS pour votre réseau (et, sous UNIX® , dans votre fichier /etc/hosts).

  7. Sélectionnez une option État pour le nouveau serveur virtuel.

    Le serveur virtuel est actif par défaut.

  8. Ne renseignez pas le champ Modules d'écoute HTTP.

    Ce champ est renseigné automatiquement lorsque vous créez un module d'écoute HTTP et l'associez à ce serveur.

    Ce champ exige que vous spécifiez un module HTTP existant. Cependant, vous ne devez pas spécifier le listener qui est utilisé par un autre serveur virtuel. Sinon, une erreur apparaît dans le journal du serveur lorsque vous redémarrez le serveur. Étant donné qu'un listener doit être associé à un serveur virtuel existant lors de sa création, tous les listeners existants sont utilisés par un autre serveur virtuel.

  9. (Facultatif) Dans la liste déroulante Module Web par défaut, sélectionnez le module devant répondre à toutes les requêtes qui ne peuvent pas être mappées vers les autres modules Web déployés sur le serveur virtuel.

    Si aucun module Web par défaut n'est spécifié, le module Web qui possède une racine contexte vide est utilisé. S'il n'existe aucun module Web avec une racine contexte vide, un module Web par défaut du système est créé et utilisé.

  10. (Facultatif) Dans le champ Fichier journal, saisissez le nom de chemin du fichier où les messages de journalisation provenant de ce serveur virtuel doivent apparaître.

    Si ce champ n'est pas renseigné, les messages de journalisation sont envoyés au journal du serveur par défaut, rép-domaine/logs/server.log.

  11. (Facultatif) Dans le champ Docroot, entrez le chemin absolu du répertoire racine de documents sur le serveur.

    Si ce champ n'est pas renseigné, rép-domaine/docroot est utilisé.

  12. Cochez la case SSO activé pour activer la connexion unique pour ce serveur virtuel.

    Cette option est désactivée par défaut. Si cette option est désactivée, les utilisateurs doivent s'authentifier séparément auprès de chaque application sur le serveur virtuel.

    La connexion unique d'une application à l'autre dans Enterprise Server est prise en charge par les servlets et les fichiers JSP. Cette fonction permet à plusieurs applications exigeant les mêmes informations de connexion utilisateur de partager ces dernières au lieu de contraindre l'utilisateur à s'authentifier séparément pour chaque application.

  13. Sélectionnez l'option Journalisation de l'accès pour activer ou désactiver la journalisation des accès.

    Sinon, vous pouvez également spécifier si la journalisation des accès est héritée du service HTTP.

  14. Dans le champ Répertoire, entrez le chemin absolu des journaux d'accès du serveur.

    La valeur par défaut est rép-domaine/logs/access.

  15. (Facultatif) Dans le champ Taille du tampon, entrez la taille du tampon, en octets.

    Une valeur inférieure ou égale à zéro désactive la mise en tampon. Si ce champ n'est pas renseigné, la taille du tampon du journal d'accès est héritée du service HTTP.

  16. (Facultatif) Dans le champ Intervalle d'écriture, entrez l'intervalle, en secondes, séparant la mise à jour des journaux d'accès.
  17. (Facultatif) Dans la section Autres propriétés, spécifiez les autres propriétés du serveur virtuel.

    Que vous spécifiiez ou non des propriétés, les propriétés par défaut docroot et accesslog du nouveau serveur sont définies sur des valeurs par défaut. La valeur par défaut de accessLoggingEnabled est false (faux) pour les profils Développeur et Cluster, true (vrai) pour le profil Entreprise. Si la valeur est true (vrai), la journalisation des accès est uniquement activée pour ce serveur virtuel. Si la valeur est false (faux), la journalisation des accès est uniquement désactivée pour ce serveur virtuel.

    Les propriétés de serveur virtuel ci-dessous sont disponibles :

    sso-max-inactive-seconds

    Indique le nombre de secondes au-delà duquel l'enregistrement de connexion unique d'un utilisateur peut être supprimé si aucune activité cliente n'est reçue. Étant donné que la connexion unique s'applique à plusieurs applications du même serveur virtuel, le fait d'accéder à une application maintient l'enregistrement de connexion unique actif.

    La valeur par défaut est de 300 secondes (5 minutes). Des valeurs supérieures offrent une persistance de connexion unique supérieure au détriment d'une plus grande consommation de mémoire sur le serveur.

    sso-reap-interval-seconds

    Indique le nombre de secondes entre les purges des enregistrements de connexion unique expirés.

    La valeur par défaut est 60.

    allowLinking

    Si la valeur est true (vrai), les ressources représentant des liens symboliques seront utilisées pour toutes les applications Web déployées sur ce serveur virtuel. Les applications Web individuelles peuvent remplacer ce paramètre en utilisant la propriété sun-web-app allowLinking du fichier sun-web.xml :

    <sun-web-app>
   <property name="allowLinking" 
      value="{true|false}"/>
</sun-web-app>

    La valeur par défaut est false.

  18. Cliquez sur OK .
