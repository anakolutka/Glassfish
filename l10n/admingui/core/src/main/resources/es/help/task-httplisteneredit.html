Para editar una escucha HTTP

  1. En el árbol de navegación, expanda el nodo Configuración.
  2. En el nodo Configuración, expanda el nodo Servicio HTTP.
  3. En el nodo Servicio HTTP, seleccione el nodo Escuchas HTTP.
  4. En la página Escuchas HTTP, haga clic en el nombre de la escucha que está editando.

    Si edita la escucha HTTP especial llamada admin-listener, deberá reiniciar el servidor para que los cambios surtan efecto. La Consola de administración no sabe que, en este caso, es necesario reiniciar el servidor.

  5. Marque la casilla de verificación Escucha activada, si desea activar esta escucha cada vez que se reinice el servidor.
  6. Marque la casilla de verificación Seguridad activada, si desea crear una escucha segura.
  7. En el campo Dirección de red, introduzca la dirección IP del servidor.

    Si la escucha se va a escuchar en todas las direcciones IP del servidor, introduzca 0.0.0.0 en el campo Dirección de red.

  8. En el campo Puerto de escucha, introduzca el valor del puerto de dicha escucha.
    • Si ha introducido 0.0.0.0 en el Paso 7, asegúrese de que el valor del puerto de escucha no se esté utilizando en ninguna dirección IP de este servidor.
    • De lo contrario, asegúrese de que el valor del puerto de la escucha no se esté utilizando en ninguna de las direcciones IP que ha especificado.
  9. En la lista desplegable Servidor virtual predeterminado, seleccione un servidor virtual.
  10. En el campo Nombre de servidor, introduzca el nombre del host que se va a utilizar en las URL que el servidor envía al cliente.

    Este nombre es el alias si el servidor hace uso de un nombre de alias. Si el servidor no utiliza un alias, deje el campo en blanco.

  11. Para redireccionar las solicitudes a otro puerto, escriba un valor en el campo Redireccionar puerto.

    Servidor empresarial redirecciona automáticamente la solicitud si se dan las dos condiciones siguientes:

    • La escucha admite otras solicitudes que no son SSL.

    • Se recibe una solicitud para la que una restricción de seguridad coincidente requiere transporte SSL.

      Servidor empresarial utiliza el número de puerto especificado en la solicitud original de forma predeterminada.

  12. En el campo Subprocesos del aceptador, introduzca el número de subprocesos de esta escucha.
  13. Desactive la casilla Funciona con habilitado para deshabilitar la inclusión del encabezado X-Powered-By: Servlet/2.4 en encabezados de respuesta HTTP generados por servlet.

    La especificación Java Servlet 2.4 define este encabezado, que los contenedores pueden agregar a respuestas generadas por servlet. De modo similar, la especificación JavaServer Pages (JSP ) 2.0 define un encabezado X-Powered-By: JSP/2.0para que se agregue (de forma opcional) a las respuestas que utilizan tecnología JSP. La inclusión del encabezado X-Powered-By: JSP/2.0 está activada de forma predeterminada para las aplicaciones web. El objetivo de estos encabezados es permitir a los administradores de sitios web recabar datos estadísticos sobre el uso de las tecnologías JSP y el servlet.

    Para obtener más información sobre cómo habilitar y deshabilitar el encabezado X-Powered-By de archivos JSP, consulte el Apéndice A, Deployment Descriptor Files de Sun GlassFish Enterprise Server v3 Prelude Application Deployment Guide.

    Los entornos de producción pueden decidir omitir la generación de encabezados X-Powered-By para ocultar su tecnología subyacente.

  14. Marque la casilla de verificación Bloqueo habilitado para habilitar el bloqueo de esta escucha y los puertos externos.
  15. (Opcional) En la sección Propiedades adicionales, especifique las propiedades adicionales.
  16. Haga clic en "Guardar".
