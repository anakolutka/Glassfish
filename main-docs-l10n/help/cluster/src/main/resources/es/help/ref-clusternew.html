Cluster nuevo

Utilice la página Cluster nuevo para crear un cluster.

La página Cluster nuevo contiene las opciones siguientes.

Nombre de cluster

Nombre del cluster.

El nombre debe cumplir los siguientes requisitos:

  • El nombre sólo puede contener caracteres ASCII.

  • El nombre debe empezar por una letra, un número o un carácter de subrayado.

  • El nombre sólo puede contener los siguientes caracteres:

    • Letras en minúscula

    • Letras en mayúscula

    • Números

    • Guión

    • Punto

    • Subrayado

  • El nombre debe ser único en el dominio y no debe ser el nombre de otro cluster, una configuración con nombre, una instancia de GlassFish Server ni otro nodo.

  • El nombre no debe ser domain, server ni otra palabra clave reservada por GlassFish Server.

Configuración

Lista desplegable de las configuraciones con nombre existentes que puede utilizar el cluster. Un cluster necesita una configuración con nombre que defina la configuración de todas las instancias que se agregan al cluster. Debe seleccionarse una configuración en la lista. El cluster utilizará la configuración seleccionada cuando se cree.

Realizar una copia de la configuración seleccionada

Si se selecciona esta opción, se copia la configuración seleccionada cuando se crea el cluster. Se asigna el nombre cluster-name-config a la copia de la configuración, donde cluster-name es el nombre del cluster del campo Nombre de cluster.

Si se selecciona default-config en la lista desplegable Configuración, debe seleccionar esta opción. La configuración default-config sólo se puede copiar, no tomar como referencia.

Hacer referencia a la configuración seleccionada

Si se selecciona esta opción, el cluster utilizará la configuración con nombre existente especificada.

Si se selecciona default-config en la lista desplegable Configuración, no debe seleccionar esta opción. La configuración default-config no puede tomarse como referencia, sino sólo copiarse.

Tipo de configuración de cluster de Message Queue

Use esta opción para especificar el tipo de cluster de broker Message Queue que se va a usar para el nuevo cluster de GlassFish Server. Por defecto, GlassFish Server usa un cluster de broker Message Queue convencional con un broker maestro, con un broker embebido en cada una de las instancias del cluster de GlassFish Server.

Si esta opción se define como Personalizada, se mostrarán varias opciones adicionales para configurar el cluster de broker Message Queue de forma que se pueda usar para el cluster de GlassFish Server. Consulte Opciones para clusters de broker Message Queue personalizados.

Instancias del servidor que se van a crear

Lista de instancias de GlassFish Server que se crearán cuando se cree el cluster. Para cada instancia, se proporciona la información siguiente:

Nombre de instancia

Nombre de la instancia.

Peso

Entero que representa el peso de equilibrio de carga de la instancia.

El peso de equilibrio de carga determina la proporción de todas las solicitudes con respecto al cluster que debe procesar la instancia.

Por ejemplo, en un cluster de dos instancias, puede necesitar que una instancia procese una solicitud de cuatro, y que otra procese las otras tres. En este caso, configure el peso de la instancia que debe procesar una de las cuatro solicitudes como 1 y el peso de la otra instancia como 3. Si prefiere usar porcentajes, defina los pesos de las instancias como 25 y 75, respectivamente.

El peso por defecto es 100.

Nodo

Lista desplegable de los nodos existentes en los que puede residir la instancia. Debe seleccionarse un nodo en la lista. La instancia residirá en el nodo seleccionado cuando se cree el cluster.

La tabla Instancias del Servidor que se van a crear también contiene las opciones siguientes.

Nuevo

Botón para crear una instancia. Al hacer clic en este botón se agrega una fila a la tabla Instancias del servidor que se van a crear.

Suprimir

Botón para suprimir una instancia.

Opciones para clusters de broker Message Queue personalizados

Cuando se defina la opción Tipo de configuración de cluster de Message Queue como Personalizada, se mostrarán las opciones siguientes para configurar el cluster de broker Message Queue de forma que se pueda usar para el cluster de GlassFish Server.

Tipo de servicio JMS

Tipo de broker (llamado host JMS en GlassFish Server) que se debe usar en el cluster de broker. El tipo seleccionado determinará los tipos de clusters de broker disponibles. Los tipos de broker disponibles son los siguientes:

Embebido

GlassFish Server gestiona el ciclo de vida y la configuración del broker. Cada instancia en cluster de GlassFish la sirve un broker que se ejecuta en la misma JVM que la instancia.

Si la opción Tipo de servicio JMS se define en Embebido, sólo se soportarán los clusters de broker Message Queue convencionales. Se mostrarán las opciones de configuración del cluster convencional.

Local

GlassFish Server gestiona el ciclo de vida y la configuración del broker. Cada instancia en cluster de GlassFish la sirve un broker que se ejecuta en una JVM distinta situada en el mismo host que la instancia.

Si la opción Tipo de servicio JMS se define en Local, se soportarán los clusters de broker Message Queue convencionales y también los mejorados (con alta disponibilidad). Se mostrarán las opciones para seleccionar el tipo de cluster y para la configuración del mismo.

Remoto

La configuración y el ciclo de vida del broker y del cluster de broker se gestionan a través de las herramientas administrativas de Message Queue.

Si la opción Tipo de servicio JMS se define en Remoto, no se mostrarán otras opciones.

Tipo de cluster de JMS

Tipo de cluster de broker Message Queue que se debe usar para el cluster de GlassFish Server.

Si la opción Tipo de cluster de JMS se define en Convencional, se mostrarán las opciones siguientes.

Tipo de almacén de configuración de JMS

Tipo de almacén de datos que se va a usar para los datos de configuración del cluster convencional. Si esta opción se define como Broker maestro, se designará un broker del cluster como broker maestro que será responsable del almacenamiento de los datos de configuración. Si esta opción se define en Base de datos compartida, los datos de configuración se almacenarán en un almacén de datos JDBC accesible para todos los brokers. En este caso, se mostrarán las opciones relacionadas con la base de datos.

Tipo de almacén de mensajes de JMS

Tipo de almacén de datos que va a usar cada broker para almacenar los datos de sus mensajes. Si se define esta opción en Archivo, cada broker almacenará los datos correspondientes en un almacén de datos basado en archivos. Si se define esta opción en JDBC, cada broker almacenará los datos correspondientes en un almacén de datos JDBC.

Si la opción Tipo de cluster de JMS se define en Mejorado (HA), se mostrarán las opciones relacionadas con las bases de datos.

Proveedor de base de datos
URL de base de datos
Usuario de base de datos

Proveedor de la base de datos, URL de acceso y usuario de la base de datos JDBC que se van a utilizar en cualquiera de estas situaciones:

  • Cuando el tipo de almacén de configuración de un cluster convencional se ha definido en Base de datos compartida.

  • Cuando el tipo de almacén de mensajes de un cluster convencional se ha definido en JDBC.

  • Cuando el tipo de cluster se ha definido en Mejorado (HA).

Para obtener información sobre proveedores soportados y los formatos de URL de acceso de cada proveedor, consulte la sección sobre persistencia basada en JDBC en Oracle GlassFish Server Message Queue Administration Guide.

Autenticación de base de datos
Contraseña de base de datos
Alias de contraseña

Información de contraseña para el usuario de la base de datos JDBC especificado en Usuario de base de datos.

Si la opción Autenticación de base de datos está definida en Contraseña, se mostrará la opción Contraseña de base de datos. Si la opción Autenticación de base de datos está definida en Alias de contraseña, se mostrará la opción Alias de contraseña.

Propiedades del cluster JMS

Lista de una o más propiedades de broker Message Queue para configurar los distintos brokers. Esta lista viene separada por dos puntos (:) y tiene este formato:

prop1Name=prop1Value:prop2Name=prop2Value:...

Si el nombre de la propiedad de broker incluye puntos, coloque dos barras invertidas antes de los puntos (\\); por ejemplo, para incluir la propiedad imq.system.max_count, especifique imq\\.system\\.max_count en la lista.
