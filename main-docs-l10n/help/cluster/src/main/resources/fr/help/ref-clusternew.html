Nouveau cluster

La page Nouveau cluster permet de créer un cluster.

Elle contient les options ci-après.

Nom du cluster

Nom du cluster.

Le nom doit respecter les exigences suivantes :

  • Il ne peut contenir que des caractères ASCII.

  • Il doit commencer par une lettre, un chiffre ou un trait de soulignement.

  • Il ne peut contenir que les caractères suivants :

    • Lettres minuscules

    • Lettres majuscules

    • Chiffres

    • Trait d'union

    • Point

    • Trait de soulignement

  • Le nom doit être unique dans le domaine et ne doit pas appartenir à un autre cluster, à une configuration nommée, à une instance GlassFish Server ou à un noeud.

  • Il ne doit pas être domain, server ou tout autre mot-clé réservé par GlassFish Server.

Configuration

Liste déroulante des configurations nommées existantes pouvant être utilisées par le cluster. Un cluster exige une configuration nommée qui définit la configuration de toutes les instances qu'il contient. Vous devez sélectionner une configuration dans la liste. Le cluster utilisera la configuration sélectionnée lors de sa création.

Créer une copie de la configuration sélectionnée

Si cette option est sélectionnée, la configuration choisie est copiée lors de la création du cluster. Le nom cluster-name-config est affecté à la copie de la configuration, cluster-name étant le nom du cluster tel que vous l'avez saisi dans le champ Nom du cluster.

Si default-config est sélectionné dans la liste déroulante Configuration, vous devez choisir cette option. La configuration default-config peut être copiée, mais pas utilisée comme référence.

Utiliser la configuration sélectionnée comme référence

Si cette option est sélectionnée, le cluster utilisera la configuration nommée existante indiquée.

Si default-config est sélectionné dans la liste déroulante Configuration, vous ne devez pas choisir cette option. La configuration default-config peut être copiée, mais pas utilisée comme référence.

Type de configuration du cluster de file d'attente des messages

Cette option permet d'indiquer le type de cluster de brokers Message Queue à utiliser pour le nouveau cluster GlassFish Server. Par défaut, GlassFish Server utilise un cluster de brokers Message Queue conventionnel avec un broker maître et avec un broker imbriqué dans chacune des instances du cluster GlassFish Server.

Si cette option a la valeur Personnalisé, plusieurs options supplémentaires sont affichées pour configurer le cluster de brokers Message Queue à utiliser pour le cluster GlassFish Server. Reportez-vous à Options pour les clusters de brokers Message Queue personnalisés.

Instances de serveur à créer

Liste des instances GlassFish Server qui seront créées en même temps que le cluster. Pour chaque instance, les informations suivantes sont fournies :

Nom d'instance

Nom de l'instance.

Poids

Entier qui représente la pondération de l'équilibrage de charge de l'instance.

La pondération de l'équilibrage de charge détermine la proportion de toutes les demandes destinées au cluster que l'instance doit traiter.

Par exemple, dans un cluster comportant deux instances, vous pouvez avoir besoin d'une instance pour le traitement de l'une des quatre demandes et d'une autre instance pour le traitement des trois demandes restantes. Dans ce cas, définissez la pondération de l'instance qui doit traiter une demande sur quatre sur 1 et celle de l'autre instance sur 3. Si vous préférez utiliser des pourcentages, définissez la pondération des instances sur 25 et 75, respectivement.

La pondération par défaut est de 100.

Noeud

Liste déroulante des noeuds existants sur lesquels l'instance peut résider. Vous devez sélectionner un noeud dans la liste. L'instance résidera sur le noeud sélectionné lors de la création du cluster.

La table Instances de serveur à créer contient également les options ci-après.

Nouveau

Bouton permettant de créer une instance. Un clic sur ce bouton permet d'ajouter une ligne à la table Instances de serveur à créer.

Supprimer

Bouton permettant de supprimer une instance.

Options pour les clusters de brokers Message Queue personnalisés

Lorsque l'option Type de configuration du cluster de file d'attente des messages a la valeur Personnalisé, les options ci-après sont affichées pour configurer le cluster de brokers Message Queue à utiliser pour le cluster GlassFish Server.

Type de service JMS

Type des brokers (appelés hôtes JMS dans GlassFish Server) à utiliser dans le cluster de brokers. Le type sélectionné détermine les types de cluster de brokers disponibles. Les types de broker sont les suivants :

Imbriqué

La configuration et le cycle de vie du broker sont gérés par GlassFish Server. Chaque instance GlassFish incluse dans un cluster est traitée par un broker exécuté dans la même JVM que l'instance.

Si l'option Type de service JMS a la valeur Imbriqué, seuls les clusters de brokers Message Queue conventionnels sont pris en charge. Les options permettant de configurer le cluster conventionnel sont affichées.

Local

La configuration et le cycle de vie du broker sont gérés par GlassFish Server. Chaque instance GlassFish incluse dans un cluster est traitée par un broker exécuté dans une JVM distincte sur le même hôte que l'instance.

Si l'option Type de service JMS a la valeur Local, les clusters de brokers Message Queue conventionnels et améliorés (haute disponibilité) sont pris en charge. Les options permettant de sélectionner le type de cluster et de configurer le cluster sont affichées.

Distant

La configuration et le cycle de vie du broker et du cluster de brokers sont gérés à l'aide des outils d'administration Message Queue.

Si l'option Type de service JMS a la valeur Distant, aucune autre option n'est affichée.

Type de cluster JMS

Type de cluster de brokers Message Queue à utiliser pour le cluster GlassFish Server.

Si l'option Type de cluster JMS a la valeur Conventionnel, les options ci-après sont affichées.

Type d'emplacement de stockage de la configuration JMS

Type de banque de données à utiliser pour les données de configuration du cluster conventionnel. Si cette option a la valeur Broker maître, un broker du cluster est désigné comme broker maître et est utilisé pour stocker les données de configuration. Si cette option a la valeur Base de données partagée, les données de configuration sont stockées dans la banque de données JDBC accessible par tous les brokers. Dans ce cas, les options associées à la base de données sont affichées.

Type d'emplacement de stockage des messages JMS

Type de banque de données que chaque broker doit utiliser pour stocker ses données de message. Si cette option a la valeur Fichier, chaque broker stocke ses données de message dans une banque de données de type fichier. Si cette option a la valeur JDBC, chaque broker stocke ses données de message dans une banque de données JDBC.

Si l'option Type de cluster JMS a la valeur Amélioré (HA), les options associées à la base de données sont affichées.

Fournisseur de base de données
URL de base de données
Utilisateur de la base de données

Fournisseur, URL d'accès et utilisateur de la base de données JDBC à utiliser dans les situations suivantes :

  • Lorsque le type d'emplacement de stockage de la configuration d'un cluster conventionnel a la valeur Base de données partagée

  • Lorsque le type d'emplacement de stockage des messages d'un cluster conventionnel a la valeur JDBC

  • Lorsque le type de cluster a la valeur Amélioré (HA)

Pour plus d'informations sur les fournisseurs pris en charge et sur les formats des URL d'accès de chaque fournisseur, reportez-vous à JDBC-Based Persistence dans le manuel Oracle GlassFish Server Message Queue Administration Guide.

Authentification de base de données
Mot de passe de base de données
Alias de mot de passe

Informations sur le mot de passe de l'utilisateur de base de données JDBC indiqué dans Utilisateur de la base de données.

Si l'option Authentification de base de données a la valeur Mot de passe, l'option Mot de passe de la base de données est affichée. Si l'option Authentification de base de données a la valeur Alias de mot de passe, l'option Alias de mot de passe est affichée.

Propriétés de cluster JMS

Liste des propriétés de broker Message Queue permettant de configurer les brokers. Les éléments de la liste sont séparés par le signe deux-points (:) et sont au format suivant :

prop1Name=prop1Value:prop2Name=prop2Value:...

Si un nom de propriété de broker contient des points, faites-les précédez de deux barres obliques inverses (\\) ; par exemple, pour inclure la propriété imq.system.max_count, indiquez imq\\.system\\.max_count dans la liste.
